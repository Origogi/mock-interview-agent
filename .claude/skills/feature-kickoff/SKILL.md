# Feature Kickoff — TPM 멀티-에이전트 디스패치 워크플로우

## 페르소나
사용자는 **TPM (Technical Program Manager)** 이다. 회사에서 진행하는 실제 업무 흐름을 본뜬 분업:

| 역할 | 담당 | 결과물 |
|------|------|--------|
| **TPM (사용자)** | 기술적 결정 + 기획 큰 그림 (목표·기술 컨벤션·UI 참조·등급 정의) | 의도와 트레이드오프 결정 |
| **PM 에이전트** (`product-manager`) | 상세 기획, 회의 어젠다, 인터페이스 합의안, 디스패치 플래닝, 백로그 등록 | 회의록 + 합의안 + 디스패치 플랜 |
| **도메인 에이전트** (`frontend-developer` / `backend-developer` / `ui-designer`) | 코드·사양서 산출 | 실제 구현물 |
| **메인 (Claude)** | 라우팅·통합·사용자 보고 (오케스트레이션 메타) | 결과 통합 + 라이브 검증 체크리스트 |

## 트리거 시그널
다음 중 하나라도 보이면 이 스킬을 발동:
- "회의 시작", "킥오프", "기획해줘", "플래닝", "큰 그림 짜줘"
- "TO 기획자/PM/디자이너/BE/FE" 같은 명시적 호명
- 새 기능 요청이 **2개 이상 도메인**(FE+BE 또는 FE+BE+디자인)을 동시에 건드림
- TPM이 큰 그림을 줬는데 디테일(위치·로딩·LangGraph 통합 등)이 비어 있음

## 워크플로우 (5단계)

### 1. 의도 파악 + 빠른 오리엔테이션
TPM이 준 큰 그림에서 추출:
- 기능 목적과 범위
- TPM이 이미 결정한 사항 (예: "v1 page3 참조", "Best/Good/Bad 3등급", "동기 호출")
- 비어 있는 디테일 (PM이 풀어 와야 하는 것)

기존 백로그(`planning/features.md`, `planning/todo.md`)와 관련 코드 위치를 빠르게 확인. **이건 메인이 해도 된다 (오리엔테이션 메타).** 단, 도메인 결정은 절대 내리지 않는다.

### 2. PM 에이전트 호출 — 상세 기획
`product-manager` 에이전트로 다음을 위임. 메인이 직접 짜지 **않는다**:
- 컨텍스트 요약 (재사용 가능 자산, 코드 인덱스)
- 회의 어젠다 (3명에게 풀어줄 질문 목록)
- **인터페이스 가합의안** — API 스펙·UI 가합의를 PM이 미리 못 박아 도메인 에이전트들이 진짜 병렬로 일할 수 있게 함
- 디스패치 플랜 (병렬 가능한 작업 분해)
- TPM에게 물을 트레이드오프 후보

PM 산출물 표준 형식:
```
1. 컨텍스트 요약 (재사용 자산)
2. 회의 어젠다 (담당자별 풀 질문)
3. 인터페이스 가합의안 (API + UI)
4. 디스패치 플랜
5. TPM 결정 필요 항목 (트레이드오프)
```

### 3. TPM 결정 받기
PM이 도출한 트레이드오프를 `AskUserQuestion`으로 TPM에게 묻는다. 보통 2–4개. PM의 추천안을 첫 옵션으로 명시.

### 4. advisor 호출 → 도메인 에이전트 병렬 디스패치
TPM 결정 직후 `advisor()` 호출 1회. 놓친 핵심 검증(회귀 위험, state 조회 경로, IME 가드 등)을 받는다.

advisor 지적을 각 도메인 brief에 반영한 뒤, `frontend-developer` / `backend-developer` / `ui-designer`를 **단일 메시지에서 병렬** 디스패치 (단, 단일 도메인 기능이면 1명만).

각 brief에 반드시 포함:
- 확정 인터페이스 합의안 (변경 X 표기)
- TPM 결정 사항
- 해당 도메인의 회귀 위험 + advisor 핵심 지적
- 산출물 형식·경로
- "Plan First 면제 — TPM 승인 완료, 즉시 실행" 명시

### 5. 통합·검수·보고
- 산출물 git diff로 핵심 부분 검수 (advisor 지적 반영 여부, 회귀 가드 유지)
- 사양 갭(디자이너 사양서 ↔ FE v0 차이 등) 발견 시 **해당 도메인 에이전트에 후속 패치 위임** (메인이 직접 패치하지 않음)
- 백로그 문서 업데이트는 PM 또는 메타로 처리
- TPM에게 결과 보고 + 라이브 검증 체크리스트

## 안티 패턴 (이전 세션에서 메인이 반복한 실수 — 절대 금지)

이 스킬이 존재하는 이유다. 같은 패턴이 보이면 **즉시 멈추고 도메인 에이전트로 위임**하라.

### ❌ 메인이 PM 역할 흡수
"라운드트립 줄이려고 메인이 어젠다·합의안을 직접 짠다"
- TPM이 "회의 시작" 또는 "기획자" 호명을 했는데도 메인이 회의 어젠다를 직접 작성
- 결과: TPM이 명시적으로 분업 패턴을 강화 지시

**왜 안 되나**: TPM이 원하는 것은 효율 최적화가 아니라 분업 구조 자체다. 라운드트립 한 번 더 일어나도 PM이 일을 한다.

### ❌ 작은 갭이라도 메인이 도메인 코드 직접 패치
"디자이너 사양서 ↔ FE 코드 갭이 단순한 시각 토큰이라 메인이 패치"
- "한 줄 패치니까 빠르게"라는 핑계로 `frontend-developer` 위임을 건너뜀

**왜 안 되나**: 한 줄 패치도 도메인 에이전트가 해야 향후 회귀 점검·토큰 일관성·관행 유지가 도메인 안에서 통제된다. 메인이 끼어들면 그 라인은 도메인 컨텍스트 밖에 위치.

### ❌ TPM이 결정해줘도 메인이 적용
"AskUserQuestion 답이 명확하니 메인이 결정 적용"
- 결정 자체는 TPM이 했지만 코드 적용을 메인이 함

**왜 안 되나**: 결정은 TPM, 적용은 도메인 에이전트. 메인은 결정을 에이전트에 전달하는 메신저일 뿐.

## Plan First 원칙 — 누가 플랜을 짜는가
- **PM이 플랜을 짠다.** 메인이 짜지 않는다.
- TPM이 PM의 플랜을 명시 승인한 뒤에야 도메인 에이전트가 코딩에 들어간다.
- 도메인 에이전트는 "Plan First 면제 — TPM 승인 완료" brief를 받고 즉시 실행 (도메인 자체 Plan First를 다시 돌리지 않음).

## 메인이 직접 처리하는 것 (메타만, 화이트리스트)
- 라우팅 결정, 에이전트 호출(병렬/순차), 결과 수집·통합·보고
- advisor 호출, AskUserQuestion
- git 작업 (commit/push), 환경 셋업·서버 기동/종료
- 자명한 메타 질의 답변 ("이 에이전트의 역할은?", 프로젝트 구조 안내)
- 빠른 오리엔테이션(파일 위치 확인, 백로그 인덱스 체크) — 단 도메인 결정은 금지

## 인터페이스 가합의안 패턴
PM이 도출하는 합의안의 표준 형식. 이게 있어야 도메인 에이전트가 진짜 병렬로 일한다.

### API 스펙 (BE↔FE)
```
[METHOD] [PATH]
Body:     { ... }
Response: { ... }
- 동기/비동기, 응답 시간 추정, 에러 코드
```

### UI 합의안 (디자이너↔FE)
```
- 위치 (composer 상단 좌측 등)
- 형태 (단일 버튼+드롭업 등)
- 동작 시퀀스 (클릭 → 호출 → disable → 채움)
- 자동 전송 여부
- 가합의 시각 토큰 (사양서 도착 전 v0용)
```

### 도메인 내부 설계 가이드라인
PM은 도메인 깊이까지 파고들지 않는다. 다만 핵심 결정(LangGraph 미사용 vs 보조 graph, IME 가드 패턴 등)은 advisor 검증 항목으로 brief에 박아 둔다. 깊은 설계는 도메인 에이전트가 자율 판단.

## 보고 형식 (TPM에게)
산출물 통합 후 다음 형식:

```
# ✅ [기능명] 완료

## 산출물
| 담당 | 결과물 | 핵심 |
|------|-------|------|
| 디자이너 | 사양서 경로 | 핵심 결정 |
| 백엔드 | 변경 파일 | 핵심 구현 (advisor 지적 반영) |
| 프론트엔드 | 변경 파일 | 핵심 구현 (회귀 가드 유지) |
| PM(통합) | 백로그 등록 | 추적 항목 |

## 인터페이스 합의안 (실제 결과 일치)
- API: ...
- UX: ...

## 남은 라이브 검증 (TPM)
- [ ] 시나리오 1
- [ ] 시나리오 2
```

## 단일 도메인 변형
TPM 요청이 1개 도메인만 건드리면 (예: BE 단독 마이그레이션, 디자이너 단독 시각 정리):
1. PM 호출은 여전히 한다 — 백로그 등록·범위 정리·테스트 시나리오는 PM 책임
2. 도메인 에이전트 1명만 디스패치
3. 안티 패턴은 동일하게 적용 — 메인은 코드 직접 작성 금지
